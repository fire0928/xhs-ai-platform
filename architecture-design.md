## 一、执行摘要

本系统是一个基于 C/S（客户端/服务器）架构的**小红书AI内容创作与自动发布平台**，支持电脑端（Electron）、鸿蒙手机端、微信小程序三端。核心挑战在于：**小红书未向普通开发者开放内容发布 API**，因此系统必须通过浏览器自动化技术实现自动发布功能。

本文档定义了系统的可扩展架构设计、关键技术决策（ADR）、长期演进路线图及风险应对策略。

---

## 二、核心痛点与技术约束

### 2.1 关键约束：无官方发布 API

| 维度 | 现状 |
|------|------|
| 小红书开放平台 | 仅面向商家/品牌方提供 Marketing API（数据读取），不提供发布接口 |
| 普通开发者 | 无法获得内容创建/发布的正式 API 权限 |
| 影响 | 自动发布必须依赖非官方渠道（浏览器自动化 / 移动端自动化） |

### 2.2 性能约束

- AI 生成响应时间 ≤ 10秒/条，批量10条 ≤ 60秒
- 多端同步延迟 ≤ 30秒
- 自动发布成功率 ≥ 99%
- 支持 1000 用户并发，单用户 1000 条内容存储

### 3.2 核心模块职责

| 模块 | 职责 | 技术栈 |
|------|------|--------|
| **API Gateway** | 统一入口、JWT认证、限流、路由 | Spring Cloud Gateway |
| **用户服务** | 注册/登录/权限管理/多端适配 | Spring Boot + JWT |
| **账号服务** | 小红书账号绑定/解绑/数据同步 | Spring Boot + AES加密 |
| **AI创作服务** | 多Agent管理/多模型调用/图文生成 | Spring Boot + HTTP Client |
| **内容审核服务** | 审核队列/敏感词检测/合规校验 | Spring Boot + DFA算法 |
| **发布引擎 (★)** | 定时调度/Playwright池化/会话管理/反检测 | Python微服务 + Playwright |
| **管理后台** | 全局用户/AI模型/内容/数据分析 | Vue3 Admin |

### 3.3 发布引擎 — 核心差异点

发布引擎是整个架构中**最关键、最复杂、最脆弱**的模块。由于没有官方API，它采用 **Playwright 浏览器自动化方案**：

```
发布队列传人(Redis Stream)
    ↓
Quartz 调度器 (秒级扫描)
    ↓
任务分发器 (按 xh_account_id 路由到对应浏览器实例)
    ↓
限流控制器 (令牌桶: 单账号 ≥ 30min间隔, 日 ≤ 5篇)
    ↓
Playwright Browser Pool (N个Chromium实例)
    ↓
┌──────────────────────────────────────────┐
│  单次发布流程:                              │
│  1. 获取已登录的浏览器实例(CDP连接)          │
│  2. 导航到 creator.xiaohongshu.com/publish │
│  3. 上传图片(1-9张, file input)            │
│  4. 填写标题 + 正文 + 话题标签              │
│  5. 点击"发布"按钮                         │
│  6. 验证发布结果 → 更新队列状态             │
└──────────────────────────────────────────┘
    ↓
反检测层: webdriver隐藏 / 随机延迟 / UA轮换
    ↓
容错层: 指数退避重试(最多3次) / 截图留存 / 失败通知
```

#### 会话管理策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **CDP 连接复用** | 连接以 `--remote-debugging-port=9222` 启动的 Chrome，复用人工登录态 | 生产环境主方案 |
| **Storage State 持久化** | 将登录状态保存为 JSON 文件，启动时加载 | 无头浏览器模式备选 |
| **Cookie 刷新机制** | 定期检查 Cookie 有效性，失效时通知用户重新授权 | 长期运行维护 |

## 四、架构决策记录 (ADRs)

### ADR-001: 采用 Playwright 浏览器自动化作为发布方案

**状态**: Accepted

**背景**: 小红书未开放发布 API 给普通开发者。

**决策**:
- 主方案: **Playwright + CDP 会话复用**
- 备选方案: Appium 移动端自动化（作为降级通道）
- 排除: 协议逆向（法律风险高、维护成本极高）

**后果**:
- ✅ 社区活跃、反检测能力强、支持池化扩展
- ⚠️ 页面结构变更可能导致选择器失效（需建立选择器版本管理）
- ⚠️ 每次发布耗时 1-2 分钟（页面加载 I/O）
- ⚠️ 存在平台风控风险（需严格控制频率）

---

### ADR-002: 发布引擎独立为 Python 微服务

**状态**: Accepted

**背景**: Playwright 生态主要在 Python/Node.js，与 Java 后端技术栈不一致。

**决策**:
- 发布引擎独立为 **Python 微服务**（FastAPI）
- 通过 **Redis Stream** 与 Java 主服务异步通信
- 发布引擎内部维护 Playwright Browser Pool

**后果**:
- ✅ 技术栈匹配，充分利用 Playwright Python 生态
- ✅ 故障隔离 — 发布引擎崩溃不影响其他业务
- ✅ 可独立扩缩容（按需增减浏览器实例数）
- ❌ 增加运维复杂度（需维护 Python 运行环境）
- ❌ 跨语言调试成本增加

---

### ADR-003: 采用微服务架构而非单体

**状态**: Accepted

**背景**: 系统涉及多端、多模型、定时任务、外部浏览器交互，复杂度较高。

**决策**:
- Phase 1 (MVP): **模块化单体**（Spring Boot 多模块，逻辑分离但同进程部署）
- Phase 2: 拆分为 **微服务集群**（用户/账号/AI/审核/发布 引擎独立部署）
- 服务间通信: 同步用 REST/Feign，异步用 Redis Stream / RabbitMQ

**后果**:
- ✅ 各模块独立开发、测试、部署
- ✅ 故障隔离、按需扩缩容
- ⚠️ 分布式事务一致性需仔细处理
- ⚠️ 初期团队若较小，建议从模块化单体起步

---

### ADR-004: 数据加密采用 AES-256-GCM

**状态**: Accepted

**背景**: 系统存储用户密码和小红书账号凭证等高敏感数据。

**决策**:
- 用户密码: **MD5 + 随机盐值**（单向哈希，无需解密）
- 小红书账号/密码: **AES-256-GCM** 加密（需要解密后传给 Playwright 登录）
- 密钥管理: KMS 或环境变量注入，禁止硬编码

**后果**:
- ✅ 即使数据库泄露，攻击者也无法直接获取明文凭证
- ⚠️ 密钥轮换机制需要在架构层面提前规划

## 七、风险矩阵与应对策略

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| **小红书前端改版导致选择器失效** | 高 | 高 | 选择器配置外置 + 版本管理 + 自动化回归测试 + 监控告警 |
| **账号被风控/封禁** | 中 | 极高 | 严格限流(日≤5篇, 间隔≥30min) / 行为随机化 / 多账号轮换 / 用户协议告知风险 |
| **Cookie/登录态频繁失效** | 高 | 中 | 会话健康检查 + 过期预警 + 快速重新授权流程 + Storage State 备选 |
| **Playwright 性能瓶颈** | 中 | 中 | Browser Pool 水平扩展 + 异步并发(同一时刻多个账号并行) + 无头模式优化 |
| **法律合规风险** | 低 | 极高 | 用户协议明确告知 / 不代存账号密码(用户自行登录授权) / 内容合规审核 |
| **AI 生成内容违规** | 中 | 高 | 敏感词系统 + AI 内容安全审核(阿里云/腾讯云内容安全API) + 人工审核兜底 |

## 十、总结与建议

### 核心观点

1. **发布引擎是整个系统的阿喀琉斯之踵** — 必须投入最优质的工程资源，建立完善的选择器版本管理、监控告警和快速修复机制。
2. **从模块化单体开始，不要一上来就上微服务** — 团队规模和业务复杂度决定了架构节奏。
3. **风控是生存红线** — 在代码层面强制限流，在产品层面充分告知用户风险，在运营层面建立应急预案。
4. **数据安全是底线** — 账号凭证加密存储，密钥管理规范化，操作审计全记录。

3. **本月**: 完成 MVP 版本的架构脚手架搭建
